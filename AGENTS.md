# AGENTS.md

Repository operating manual for any coding agent or tool working in this
checkout. This file is tool-neutral: it does not assume a specific assistant,
IDE, or private orchestration layer.

## What this repository is

IRIN is a local-first, single-operator product. It is a
Git mono-repo assembled from three subtrees, each independently documented:

| Path | Responsibility | Docs |
| --- | --- | --- |
| `council-rs/` | Council deliberation engine, provider adapters, War Room Web, and the Tauri desktop shell | [`council-rs/README.md`](council-rs/README.md) |
| `gateway/` | Optional governed routing, metering, watch plane, signing, and the durable directive outbox | [`gateway/README.md`](gateway/README.md) |
| `sentinel/` | Deterministic watch contract and shared `sovereign-protocol` wire types | [`sentinel/README.md`](sentinel/README.md) |

See [`docs/architecture.md`](docs/architecture.md) for how the pieces fit
together, and [`docs/security-claims-vs-reality.md`](docs/security-claims-vs-reality.md)
for the current, precise security boundary.

## Source, configuration, and runtime state are separate

- **Source** is this Git checkout. It is the only thing that should be edited
  by an agent.
- **Configuration** (`~/.config/irin/gateway.env`, `~/.irin/ledger_key.pem`,
  provider credentials in the operator's login shell) is private, generated or
  operator-owned, and lives outside the repository.
- **Runtime state** (Docker volumes, `~/.local/state/irin/`, `sessions/`,
  `runs/`, `librarian_chats/`) is generated by running the product and is not
  committed.

Never read credential-bearing configuration: environment files, shell
profiles, tokens, API keys, or signing keys are not model context. When a
diagnostic task requires runtime evidence, inspect only the bounded,
non-secret status, receipt, or log needed for that task; do not bulk-read
session content or publish local runtime data. Nothing from configuration or
runtime state belongs in a commit. Provider credentials come only from the
operator's login-shell environment; IRIN does not import or persist them, and
neither should a change you make.

## Safe commands

From the repository root:

```bash
make setup           # macOS full-stack config + managed runtime
make warroom         # macOS/Ubuntu Council + browser War Room
make runtime-status   # liveness and source-identity check
make verify           # isolated $0 signed-directive proof, no provider keys
make verify-down      # tear down the isolated verification stack
make build            # cargo build --workspace --release
make test             # cargo test --workspace
make release-check    # release-tree completeness and hygiene guard
```

## Worktree and shipping lifecycle

Every non-trivial change follows one executable path:

```bash
make worktree BRANCH=fix/example
make preflight
make check
make ship-check
```

Use one named branch and one owner per worktree. Do not ship from `main`, a
detached checkout, or a branch whose recorded `origin/main` base has moved.
After another branch merges, update the remaining branch and rerun
`make ship-check`. A feature-specific test is not a completion claim: the ship
receipt must include every selected product lane, including the embedded and
native Tauri surfaces for a War Room Web change. See
[`docs/development-workflow.md`](docs/development-workflow.md).

Gortex is the primary code-intelligence surface for the operator workflow.
Agents must try its MCP tools first: `smart_context` at task entry,
`get_editing_context` before non-trivial edits, and `detect_changes` plus
affected-test analysis before completion. The daemon is expected to remain up.
If a client fails to mount or discover the MCP, report that integration failure
and immediately use `gortex call`/the dedicated CLI through
`scripts/gortex-worktree.sh`; do not silently skip the graph pass. Public
contributors without Gortex use source reads and the same deterministic test
gates. Gortex never replaces source inspection or tests.

`make verify` is the fastest way to prove a change did not break the
Sentinel-to-signed-directive path without spending on a provider call.
`make setup` and `make runtime-*` are macOS-only and start real local services
(Docker, ports, launchd). `make warroom` is the macOS/Ubuntu browser-development
path and starts Council plus Next.js in the foreground. Run none of them as a
routine check during a documentation or source change.

## Proportionate tests

Match the test to the change:

- Rust source under `council-rs/`, `gateway/`, or `sentinel/`: `cargo test`
  scoped to the crate you touched, then `make test` for the full workspace
  before a PR.
- War Room Web (`council-rs/warroom/web/`): `npm run lint`, `npm run
  typecheck`, `npm test` from that directory, or `make -C council-rs
  warroom-check` from the repository root for the full frontend gate.
- Gateway signing, watch, or ledger paths: the crate-local `cargo test`
  suite plus `make verify` for the end-to-end no-key proof.
- Documentation-only changes: `git diff --check` for whitespace, a link
  check if available, and `make release-check`.

Do not run a live provider smoke call (`--smoke-provider`, `--smoke-model`),
arm the watch producer, or enable `WATCH_PRODUCER_ENABLED` as part of routine
verification. Those paths can incur real cost or take a real action and are
opt-in, documented separately in `gateway/docs/runbooks/arming-authorization.md`.

## Secrets

Never hardcode a provider API key, Gateway caller key, admin token, or
signing-key value in source, tests, fixtures, or documentation. Use an
obvious placeholder (`<YOUR_API_KEY>`) in examples. Test fixtures that
intentionally contain realistic-looking fake credentials for redaction
testing are allow-listed in `.gitleaks.toml`; do not add a new
realistic-looking value outside that allow-list.

## Contribution boundary

- One logical change per PR; keep unrelated cleanup out.
- Changes under `sentinel/sovereign-protocol/**`, `**/comms*`, escalation or
  directive envelope code, the outbox, capability-token code, or
  `.github/workflows/**` carry signing, wire-shape, or CI authority and get a
  heavier review pass regardless of size.
- PR titles, descriptions, and commit messages are part of the public
  project record: describe the shipped behavior, not the private process
  used to produce it. See [`CONTRIBUTING.md`](CONTRIBUTING.md) for the full
  public-language and commit-message conventions, and
  [`scripts/check-public-pr-language.sh`](scripts/check-public-pr-language.sh)
  for the enforced patterns.
- Do not commit anything under a private agent's own instruction, plan, or
  session-history files; `scripts/check-release-tree.sh` rejects nested
  `AGENTS.md`/`CLAUDE.md`/`RTK.md` and similar private-agent paths from a
  release tree.

Report security vulnerabilities per [`SECURITY.md`](SECURITY.md); do not open
a public issue for one.
